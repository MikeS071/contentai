# ca-04: Template Engine — Embedded Prompts with Local Override

## Context
You are building the template engine for **ContentAI** — a Go CLI tool. All LLM prompts ship embedded in the binary but can be overridden by local files in `content/templates/`.

Read `SPEC.md` in the repo root for full project context.

## Deliverables

### 1. Package: `internal/templates/`

```
internal/templates/
  engine.go          # Template resolution + variable substitution
  embedded.go        # go:embed directives
  templates_test.go  # Tests
templates/                  # Source templates (embedded via go:embed)
  perspective-architect.md
  voice-extractor.md
  deep-post-ideas.md
  creative-thought-partner.md
  blog-writer.md
  qa-checklist.md
  social-copy.md
  hero-prompt.md
```

### 2. Embedded Templates

Use `//go:embed` to bundle all template files into the binary:

```go
//go:embed templates/*.md
var embeddedTemplates embed.FS
```

### 3. Template Engine

```go
type Engine struct {
    contentDir string // for local override resolution
}

func NewEngine(contentDir string) *Engine

// Get resolves a template by name:
// 1. Check content/templates/<name>.md (local override)
// 2. Fall back to embedded templates/<name>.md
// Returns the template content as string
func (e *Engine) Get(name string) (string, error)

// GetWithVars resolves template + substitutes variables
// Variables use {{.VarName}} syntax (Go text/template)
func (e *Engine) GetWithVars(name string, vars map[string]interface{}) (string, error)

// List returns all available template names
func (e *Engine) List() []string

// Export writes all embedded templates to content/templates/ for customization
func (e *Engine) Export(destDir string) error

// IsOverridden checks if a template has a local override
func (e *Engine) IsOverridden(name string) bool
```

### 4. Template Contents

Write each template based on the SPEC.md descriptions. These are the actual prompts sent to LLMs:

**`perspective-architect.md`** — The 5-phase guided discovery prompt for generating voice profiles. Full content is provided in SPEC.md under "contentai init". This is a comprehensive multi-phase prompt that guides a user from "I don't know what I stand for" to a complete intellectual blueprint. Include all 5 phases: Excavation, Research & Expansion, Synthesis & Mission Emergence, Core Idea Crystallization, The Blueprint. Output format should produce a structured markdown blueprint.

**`voice-extractor.md`** — Analyzes example articles to extract voice patterns:
```
You are analyzing writing samples to extract a comprehensive voice profile.

Given the following example articles and intellectual blueprint, create a voice profile document that captures:

## Mission
{{.Mission}}

## Voice & Tone Rules
- How sentences open (numbers, names, moments — not generic statements)
- Paragraph and section rhythm
- Use of named insights / signature phrases
- What to avoid (AI-isms, bullet lists in narrative, restating thesis)

## Style Benchmarks
Analyze the examples and identify:
- What makes them distinctive
- Patterns in structure
- Language choices that recur
- The "test" — could anyone else have written this?

## Non-Starters
- Patterns to never use
- Tones to avoid
- Structures that don't fit

## Example Articles
{{.Examples}}

## Blueprint
{{.Blueprint}}
```

**`deep-post-ideas.md`** — Extract 5 structured post outlines from reference material. Include the language techniques (second-person, imperative verbs, absolutes, opposing pairs). Include example phrasing for reference.

**`creative-thought-partner.md`** — The insight-mining conversation prompt with four breakthrough drivers (pattern spotting, paradox hunting, naming the unnamed, contrast creation). Include the flow guidelines and constraints.

**`blog-writer.md`** — Article generation from insights + voice profile:
```
You are writing an article based on insights from a creative exploration session.

## Voice Profile
{{.VoiceProfile}}

## Blueprint Core Ideas
{{.CoreIdeas}}

## Post Outline / Topic
{{.Outline}}

## Source Insights
{{.Insights}}

## Instructions
Write a 500-1000 word article in Markdown.
- Open with a number, a name, or a moment. Not a trend statement.
- One thesis. One story arc.
- Sections are short: heading + 2-4 paragraphs of 1-4 sentences.
- No bullet points in the narrative.
- Name the insight — produce a phrase the reader will repeat.
- Say it once. Don't restate.
- Earn the ending — natural landing, not a recap.
- Write like a person. Read every sentence aloud.
```

**`qa-checklist.md`** — QA prompt with 7 checks:
```
You are a content QA reviewer. Analyze the article against these checks and the voice profile.

## Article
{{.Article}}

## Voice Profile
{{.VoiceProfile}}

## Checks
1. **no_secrets**: Scan for API keys, tokens, passwords, internal URLs, IP addresses
2. **voice_consistency**: Compare tone against voice profile benchmarks — flag passages that sound generic or robotic
3. **accuracy**: Flag claims presented as facts without evidence or hedging
4. **dash_cleanup**: Find unnecessary mid-sentence dashes/hyphens that break grammar
5. **dedup**: Flag repeated ideas or sentences (even if phrased differently)
6. **reading_level**: Flag overly complex or robotic passages
7. **length**: Warn if outside 500-1000 word range

For each issue found, provide:
- Check name
- The problematic passage (exact quote)
- Why it's an issue
- A rewritten version that fixes the issue

Output as JSON array of check results.
```

### 5. CLI Command

**`contentai templates export [--dir content/templates]`**
- Writes all embedded templates to the specified directory
- Warns if files already exist (use `--force` to overwrite)
- Prints list of exported files

Register as cobra command in `cmd/templates.go`.

### 6. Tests

Write tests FIRST:

- `TestGetEmbedded` — Get("blog-writer") returns embedded template content
- `TestGetLocalOverride` — local file exists → returned instead of embedded
- `TestGetNotFound` — Get("nonexistent") → error
- `TestGetWithVars` — template with {{.Title}} → substituted correctly
- `TestGetWithVarsMissing` — missing variable → error or empty string (decide)
- `TestList` — returns all 8 template names
- `TestExport` — writes all templates to temp dir, verify file count and content
- `TestExportNoOverwrite` — existing file → skipped (no --force)
- `TestExportForce` — existing file + force → overwritten
- `TestIsOverridden` — true when local exists, false when not
- `TestEmbeddedTemplatesExist` — all 8 expected templates are embedded
- `TestTemplateContentNotEmpty` — each template has >100 chars of content

Target: 85% coverage on `internal/templates/`.

## Constraints

- Use `text/template` (stdlib) for variable substitution
- No external deps beyond stdlib
- Template files must contain the actual prompt content (not placeholders)
- Files must be ≤400 lines each
- All tests must pass before committing
