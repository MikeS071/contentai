# ContentAI — Specification

> Reusable Go CLI + OpenClaw skill for AI-powered content creation, publishing, and social scheduling.

## Overview

ContentAI is a command-line tool that orchestrates the full content lifecycle: idea generation from knowledge bases and conversation history, AI-assisted drafting with personalized voice, automated QA with auto-fix, hero image generation, publishing to any CMS endpoint, and social media scheduling with pluggable platform adapters.

It ships as a standalone Go binary and an OpenClaw skill. The CLI is self-contained (own LLM access, own config) but integrates deeply with OpenClaw for conversation history, channel context, and agent orchestration.

## Architecture

```
contentai CLI (Go binary)
├── init          → Perspective Architect → voice.md + blueprint
├── kb            → Manage influential blogs/RSS knowledge base
├── ideas         → Deep Post Ideas prompt → ranked outlines from KB + conversations
├── new           → Create content/<slug>/ scaffold
├── draft         → Creative Thought Partner prompt → article.md
├── qa            → QA checklist + auto-fix → diff for approval
├── hero          → Image API + palette rotation + text overlay
├── publish       → Pluggable HTTP adapter → CMS endpoint
├── schedule      → Pluggable social adapters → posting calendar
├── post          → Fire due scheduled posts (cron-called)
└── list          → Show all content items + lifecycle state
```

## Commands

### `contentai init [name]`

Interactive onboarding wizard:

1. Creates `contentai.toml` with sensible defaults
2. Prompts: "Paste 2-3 of your best articles" → saves to `content/examples/`
3. Runs **Perspective Architect** prompt (5-phase guided discovery):
   - Phase 1: Excavation — surfaces influences, problems, recurring explanations
   - Phase 2: Research — deepens understanding of key thinkers (web search)
   - Phase 3: Synthesis — connects dots, mission emerges
   - Phase 4: Crystallization — identifies 3-5 "book-worthy" core ideas
   - Phase 5: Blueprint — produces `content/blueprint.md`
4. Analyzes example articles + blueprint → generates `content/voice.md`
5. Prompts for LLM API key, image API key, publish endpoint, social accounts
6. Prompts: "Add RSS feeds of blogs you follow" → saves to `content/kb/feeds.toml`

The Perspective Architect prompt ships at `templates/perspective-architect.md` and is customizable.

### `contentai kb`

Manage the knowledge base of influential content.

```bash
contentai kb add-feed https://blog.example.com/rss    # add RSS feed
contentai kb add-feed --opml feeds.opml                # import OPML
contentai kb list-feeds                                 # show tracked feeds
contentai kb sync                                       # fetch latest from all feeds
contentai kb add-note path/to/transcript.md            # add raw material
contentai kb search "topic"                             # semantic search across KB
```

KB content is cached locally as markdown in `content/kb/blogs/` and `content/kb/notes/`. Sync fetches new posts since last check. Semantic search uses LLM embeddings (or simple keyword matching as fallback).

### `contentai ideas [--from-kb] [--from-conversations] [--count 5]`

Generate article ideas using the **Deep Post Ideas** prompt.

Sources (combined):
- KB articles (RSS feed content, synced blogs)
- OpenClaw conversation history (if `openclaw.channel_history = true`)
- User notes in `content/kb/notes/`
- Existing blueprint core ideas + content pathways

Process:
1. Gathers recent KB content + conversation snippets
2. Sends to LLM with Deep Post Ideas prompt + voice profile + blueprint
3. Generates N structured outlines (core paradox, transformation arc, key examples, actionable steps)
4. Saves to `content/ideas/<date>-batch.md`
5. Presents interactively: user picks which ideas to develop

Each idea outline includes enough structure to seed a `contentai draft`.

### `contentai new <slug> [--from-idea N] [--title "..."]`

Create a new content item directory:
```
content/<slug>/
├── meta.json    (status: "draft", title, slug, created_at)
├── article.md   (empty or seeded from idea outline)
└── social.json  (empty templates)
```

If `--from-idea N` is provided, seeds `article.md` with the outline from the ideas batch.

### `contentai draft <slug> [--source file.md] [--interactive]`

Generate or refine an article draft.

**Context assembled for LLM call:**
- `content/voice.md` — brand voice profile
- `content/blueprint.md` — intellectual blueprint (core ideas, mission, content pathways)
- `content/examples/*.md` — user's example articles (style reference)
- `content/<slug>/meta.json` — title, topic context
- `content/<slug>/article.md` — existing draft (if refining)
- `--source` material — additional reference content (transcript, notes, URL)
- OpenClaw conversation history (if enabled) — recent relevant exchanges

**Prompt chain:**
1. Creative Thought Partner prompt runs first — mines source material for breakthrough insights using four drivers (pattern spotting, paradox hunting, naming the unnamed, contrast creation)
2. Blog Writer prompt takes the insights + voice profile → produces 500-1000 word article
3. Article saved to `content/<slug>/article.md`

`--interactive` mode: CLI engages in back-and-forth conversation to mine insights before writing (like the Creative Thought Partner's question flow).

### `contentai qa <slug>`

Run automated QA + auto-fix.

**Process:**
1. Loads `content/<slug>/article.md` + `content/voice.md`
2. Runs all QA checks (built-in + custom from `qa.rules_file`):
   - **no_secrets**: regex scan for API keys, tokens, passwords, internal URLs
   - **voice_consistency**: LLM compares article tone against voice.md benchmarks
   - **accuracy**: LLM flags claims without evidence or hedging
   - **dash_cleanup**: detect unnecessary mid-sentence dashes/hyphens
   - **dedup**: flag repeated ideas/sentences
   - **reading_level**: flag overly complex or robotic passages
   - **length**: warn outside 500-1000 word range
3. For each issue found:
   - Describes the problem
   - If `auto_fix = true`: generates rewritten passage, presents as diff
4. User reviews: approve all / approve individual / reject
5. Approved fixes applied to `article.md`
6. Results saved to `content/<slug>/qa.json`
7. `meta.json` status updated: `draft` → `qa_passed`

### `contentai hero <slug> [--no-title] [--palette N]`

Generate hero images.

**Process:**
1. Reads `meta.json` for title, selects palette (hash of slug mod 8, or `--palette N`)
2. Calls image API (DALL-E 3 default) with prompt:
   - Abstract/editorial composition using palette colors
   - Topic-informed but not literal (no faces, no text in image)
3. If `title_overlay = true` and not `--no-title`: overlays title text on image (Go image/draw or Pillow via exec)
4. Generates two variants:
   - `hero.png` — standard (1792x1024)
   - `hero-linkedin.png` — cropped/resized to 1200x627
5. Saves to `content/<slug>/`

**Built-in palettes** (8, rotated by slug hash):
1. Deep ocean blues + electric cyan on dark navy
2. Warm amber + burnt orange on charcoal
3. Forest green + gold accents on dark slate
4. Purple gradient + magenta highlights on midnight
5. Coral pink + soft cream on dark rose
6. Steel blue + silver on gunmetal
7. Emerald + lime accents on dark forest
8. Sunset gradient (orange→pink→purple) on deep indigo

### `contentai publish <slug> [--approve] [--dry-run]`

Publish article to configured endpoint.

**Gates:**
1. If `qa_gate = true`: requires `meta.json` status ≥ `qa_passed`
2. If `default_publish_gate = true`: requires `--approve` flag (explicit intent)

**Process:**
1. Validates: article exists, QA passed (if gated), hero images exist
2. If `publish.type = "http"`: POSTs to configured URL with field mapping
3. If `publish.type = "static"`: copies to output directory
4. Updates `meta.json`: status → `published`, published_at timestamp
5. Triggers newsletter notification if configured (v2)

`--dry-run`: shows what would be sent without sending.

### `contentai social <slug> [--approve]`

Generate social media copy.

**Process:**
1. Reads article + voice profile + blueprint
2. LLM generates platform-specific copy:
   - X: ≤280 chars, hook + link + optional image URL
   - LinkedIn: longer-form, professional tone, 1-3 paragraphs + link
3. Saves to `content/<slug>/social.json`
4. Presents for review: user edits or approves

### `contentai schedule <slug> [--approve] [--immediate]`

Schedule or immediately post social content.

**Gates:**
1. Requires `meta.json` status = `published`
2. If `default_social_gate = true`: requires `--approve`
3. **Hard rule: no auto-posting without explicit approval. Ever. Not configurable.**

`--immediate`: posts now instead of scheduling.

**Scheduling:**
1. Finds next available slot per `[schedule]` config (weekday, time window)
2. Writes slot to `content/posting-calendar.json`
3. Cron calls `contentai post --check` to fire due posts

### `contentai post --check`

Fire due scheduled posts. Called by cron.

**Process:**
1. Reads `posting-calendar.json`
2. For each due slot:
   - Preflight: validate image URLs reachable (HEAD request)
   - Post to each enabled platform via adapter
   - Update slot status: `scheduled` → `posted` (or `failed` with reason)
3. Log results

### `contentai list [--status draft|published|...] [--json]`

Show all content items with lifecycle state.

## Social Adapters

Interface:
```go
type SocialAdapter interface {
    Post(ctx context.Context, text string, imageURL string) (postURL string, err error)
    Validate() error
    Name() string
    MaxLength() int
}
```

Shipped adapters: **X** (OAuth 1.0a), **LinkedIn** (OAuth 2.0 bearer).

Adding new adapters: implement the interface in `internal/social/`, register in adapter map. Community can add Bluesky, Mastodon, Threads, etc.

## Publisher Adapters

Interface:
```go
type Publisher interface {
    Publish(ctx context.Context, item ContentItem) (url string, err error)
    Validate() error
    Name() string
}
```

Shipped adapters: **HTTP** (generic REST API with configurable field mapping), **Static** (write files to directory).

## OpenClaw Integration

When installed as an OpenClaw skill:

1. **Conversation mining**: CLI reads OpenClaw channel history to find idea-worthy moments
2. **Memory search**: queries memd / MEMORY.md for relevant context during drafting
3. **Agent orchestration**: the OpenClaw agent can run the full pipeline, presenting approvals to the user via chat
4. **Skill file**: teaches the agent when/how to use each command
5. **Workspace files**: AGENTS.md snippet (usage rules), TOOLS.md snippet (config), MEMORY.md entry

The agent workflow when user says "write an article about X":
1. `contentai ideas --from-conversations` (or `contentai new <slug> --title "X"`)
2. `contentai draft <slug> --source <context>`
3. Present draft to user → iterate
4. `contentai qa <slug>` → show diff → user approves
5. `contentai hero <slug>`
6. `contentai publish <slug>` → ask user for `--approve`
7. `contentai social <slug>` → show copy → user approves
8. `contentai schedule <slug> --approve`

## Non-Goals (v1)

- Newsletter / email notifications (separate concern, v2)
- Web UI / dashboard (users build their own, CLI is the interface)
- Multi-user / team workflows (single-user CLI)
- Auto-posting without approval (hard rule, never)
- Video content generation
- Podcast show notes (could be added as template)

### AGENTS-SNIPPET.md

Appended to `~/.openclaw/workspace/AGENTS.md` during `install.sh --openclaw`:

```markdown
## ContentAI (Go CLI — `contentai`)
- **Binary:** `contentai` — AI-powered content pipeline: ideas → draft → QA → hero → publish → social.
- **Skill:** `contentai` — read SKILL.md for full command reference.

### When to use ContentAI
- User asks to write a blog post, article, or insight
- User wants article ideas or content suggestions
- User says "publish" or "schedule" for an existing draft
- Morning content scan / idea generation routine
- **DO NOT use for:** social-only posts (no article), quick one-line responses, non-content tasks

### How to use it
1. **Generate ideas:** `contentai ideas --from-kb --from-conversations --count 5` → present to user
2. **New article:** `contentai new <slug> --title "..."` (or `--from-idea N`)
3. **Draft:** `contentai draft <slug>` → present draft → iterate with user feedback
4. **QA:** `contentai qa <slug>` → show diff of auto-fixes → user approves
5. **Hero image:** `contentai hero <slug>`
6. **Publish:** ask user for explicit approval → `contentai publish <slug> --approve`
7. **Social copy:** `contentai social <slug>` → present copy → user approves
8. **Schedule:** ask user for explicit approval → `contentai schedule <slug> --approve`

### Hard rules
- **Never publish or schedule without explicit user approval.** Always ask first.
- **Always run QA before presenting to user.** Don't show raw drafts.
- **Include voice.md + blueprint.md context in all content LLM calls.**
- **Present ideas, don't auto-execute.** User picks which ideas to develop.
```

### install.sh --openclaw

The installer (same pattern as agent-swarm):

1. Install binary via `go install`
2. Copy all `skill/` files to `~/.openclaw/workspace/skills/contentai/`
3. Append AGENTS-SNIPPET.md to AGENTS.md (idempotent check)
4. Append TOOLS-SNIPPET.md to TOOLS.md (idempotent check)
5. Append MEMORY-ENTRY.md to MEMORY.md (idempotent check)
6. Create `content/` directory structure
7. Install cron: `*/5 * * * * contentai post --check >> logs/contentai-post.log 2>&1`
8. Print next steps: run `contentai init` to set up voice profile
